# ADR-017: Cash-Dividend Read Model — Events Are Reconciliation, Ledger Is Truth

**Status:** Accepted
**Date:** 2026-04-22
**Context:** Phase A of the corporate-actions roadmap ships cash dividends (EODHD `/api/div/{SYMBOL}` ingest, a "Dividends Received" column on Holdings, a "Dividend Income" KPI on the dashboard, anomaly reconciliation). Splits (ADR-014) established the `corporate_action_*` events-table pattern for share-count arithmetic: the ledger holds raw broker data, the events table is a read-time multiplier. Cash dividends look superficially identical — same backfill shape, same idempotency constraints, same admin surface — but they differ in one decisive way: the broker already books the cash receipt as a `DIVIDEND` row in `transactions`. So two candidates exist for "source of truth" about user dividend income: (a) the declared-events table via `qty × amountPerShare`, (b) the transactions ledger. Picking wrong creates double-counting, tax-reporting drift, and a surface where a user can see different dividend totals on different pages of the same app. This ADR records the choice and the reasoning behind it.

---

## Decision 1: Analytics Derive Exclusively From `transactions`; Events Are Reconciliation-Only

**Choice:** A new `corporate_action_cash_dividend` table stores one row per declared dividend event, keyed by canonical `security_master_id`, with `(security_master_id, ex_date)` uniquely constrained, a positive-amount `CHECK`, and a 3-letter currency `CHECK`. It feeds **exactly one consumer**: `DividendAnomalyService`, an admin-only diagnostic that flags declarations on held positions with no matching broker DIVIDEND transaction within a tolerance window. Every user-visible dividend number — `HoldingDto.totalDividendsReporting`, `HoldingDto.totalDividendsYtdReporting`, `HoldingDto.dividendYieldOnCostPct`, and the per-portfolio dashboard `totalDividendIncome{Lifetime,Ytd}` — is computed by `DividendAnalyticsService.aggregateByListing(portfolioId, reportingCurrency)`, which issues one native-SQL `SUM(normalized_reporting_amount)` against `transactions` filtered by `transaction_type = 'DIVIDEND'` and `reporting_currency_at_normalization = :reportingCurrency`. `RealizedGainCalculator` already treats DIVIDEND as a no-op in the AVCO walk, so realized P/L stays untouched.

**Rationale:** The **Accounting Tie-Out Principle** (SYSTEM_SNAPSHOT §4) requires that every row sums to the summary header *and* reconciles to what the broker actually paid. If analytics read `qty × declared amount` from the events table, Holdings would show numbers that don't match the user's 1099-DIV / T212 statement — fractional-share thresholds, withholding tax, currency conversion losses, and DRIP reinvestment all make declared-vs-received divergence routine. Reading from `transactions` means Holdings, KPI, yield-on-cost, and future tax exports all emit the same value, because they all pull from the same immutable ledger row with its `normalized_reporting_amount` frozen at transaction-date FX (ADR-011 principle). A user cross-checking any number against any external source sees the same number the system was given.

**Why the events table exists at all:** Reconciliation. The broker feed *does* miss payments — the observed Trading212 Phase-A case: 31 MSFT/MA declarations on fractional-share holdings with no corresponding DIVIDEND row, concentrated below a $2 expected-total threshold. Without a declared-events table, a silent broker-feed gap is indistinguishable from a correctly-zero position. With it, `GET /api/admin/corporate-actions/dividends/anomalies?portfolioId=…&minExpectedTotal=…` answers "which declarations on held positions have no matching broker transaction?" — advisory only; the user decides whether to manually book the missing DIVIDEND or treat the row as broker-correct noise.

**Why not cache the declared amount back into `transactions`:** Because that would make the ledger lie. The ledger records what the broker *reported*, not what the company *declared*. A company-declared $10 dividend paid net-of-15%-tax as $8.50 lands in the ledger as $8.50, and that's the number a bank statement, a tax form, or a SnapTrade audit will agree with. Back-writing the declared $10 into a transaction row destroys the audit trail — the same reasoning that forbids ledger mutation for stock splits in ADR-014 §Decision 1.

## Decision 2: Analytics FX Is Historical, Not Live

**Choice:** `DividendAnalyticsService` does not call `CurrencyConversionService` or `ExchangeRateService`. It sums the already-normalized `transactions.normalized_reporting_amount` column, populated at sync time by `TransactionFxNormalizer` using the FX rate in effect on that transaction's `transaction_date`. Transactions whose `reporting_currency_at_normalization` doesn't match the requested currency are excluded from the sum — consistent with the existing `TransactionRepository.findAvgReportingPriceByPortfolio` discipline.

**Rationale:** A dividend received on 2023-02-15 was worth what it was worth on 2023-02-15. Reporting it today at live FX overstates or understates what the user actually received by years of currency drift. Historical FX per transaction also means changing the user's reporting currency triggers the same `normalized_reporting_*` rebuild as every other ledger-projection field — a single unified cache invalidation, not a second one-off for dividends (ADR-016 §Session-Scoped principle). Skipping rows that haven't yet been renormalized is correct, not a bug: they'll reappear once the sweeper catches them up.

## Decision 3: Anomaly Detection Is Advisory and Admin-Gated in Phase A

**Choice:** `DividendAnomalyService.findAnomalies(portfolioId, minExpectedTotal?)` is the only consumer of the events table. It iterates active holdings via `HoldingRepository.findActiveWithSecurityAndListingsByPortfolioId`, walks declared events older than 30 days (to let the broker feed catch up), computes `qtyHeldAtExDate` from ledger BUY/SELL rows, and checks for a matching DIVIDEND transaction within ±10 business days of `pay_date` (or `ex_date` → `ex_date + 30 days` when `pay_date` is null). A row emits only when qty > 0 and no match exists. `minExpectedTotal` suppresses sub-threshold rows — the observed Trading212 signal was that all 31 flagged rows were under $2, essentially fractional-share withholding. No user-facing UI: `CorporateActionDividendAdminController` requires superuser; the endpoint is for auditing broker-feed reliability, not for direct user consumption.

**Rationale:** The signal is real but noisy; surfacing it to end users without a triage workflow (dismiss-as-noise, mark-as-paid, auto-manual-insert) would bury the real gaps in fractional-share false positives. Shipping admin-only in Phase A means the mechanism is observable and tunable against live data before we commit to a user-facing workflow in Phase B. Keeping it read-only — the service never writes to `transactions` on behalf of the user — preserves the invariant that the ledger is only ever mutated by broker sync or a user-initiated transaction action.

---

## Consequences

- **Single source of truth for user dividend income:** the `transactions` table. Holdings column, Dashboard KPI, yield-on-cost, and future tax exports all resolve to the same `normalized_reporting_amount` sums. Tie-out across surfaces is architectural, not a test-suite obligation.
- **Changing reporting currency recomputes analytics automatically** via the existing `normalized_reporting_*` rebuild path — no dividend-specific migration needed. The events table stores its declared currency untouched.
- **Deleting all rows from `corporate_action_cash_dividend` would not change a single number on any user-facing screen** — only the anomaly report would go silent. The events table is structurally non-critical; it is purely an audit aid. This is the inverse of `corporate_action_split`, whose deletion breaks realized P/L.
- **Phase-B mergers and spin-offs inherit the read-model split.** A merger's cash component lives in `transactions` as the broker-reported forced liquidation; `corporate_action_merger` (when it lands) holds the ratio + cash-per-share metadata and feeds AVCO state transitions only, never user cash-flow totals.
- **Fractional-share / withholding / DRIP edge cases are features, not bugs.** The anomaly endpoint surfaces them as advisory signal (filterable via `minExpectedTotal`); the analytics layer sees the broker's number and is internally consistent with every statement the user can cross-check.
- **Audit trail is preserved end-to-end.** The ledger rows are broker-reported-verbatim, the events table rows record the provider's declaration verbatim (`source = 'EODHD' | 'MANUAL'`), and reconciliation is computed at read time — no mutation, no back-write, no migration required to correct either side.
