# ADR-014: Read-Time Application of Corporate Actions (Stock Splits)

**Status:** Accepted
**Date:** 2026-04-20
**Context:** AVCO realized-P/L was structurally wrong for any security that underwent a stock split. Concretely, the Trading212 AMZN case: a pre-split BUY of `1 @ 2000` followed by a post-split SELL of `20 @ 150` produced a phantom ≈ $5,000 loss, because `AvcoState` held `totalShares=1, totalCost=2000` at the moment 20 shares were sold at $150 — an arithmetic impossibility the engine was willing to execute. Two shapes of fix exist: (a) rewrite the transaction ledger on every split (back-adjust pre-split qty/price rows), or (b) leave the ledger untouched and apply splits at read time inside the AVCO state machine. Both eliminate the phantom loss; they differ in what they do to the audit trail, the migration posture, and the cost of correcting a bad split feed. Stock dividends (e.g. a 5% bonus expressed as `21:20`) fold into the same math and must be handled by the same path or they become a second divergent story.

---

## Decision 1: Splits Are a Read-Time Multiplier, Not a Ledger Mutation

**Choice:** A new `corporate_action_split` table stores one row per split/stock-dividend event, keyed by canonical `security_master_id`, with `(security_master_id, ex_date)` uniquely constrained and a positive-ratio `CHECK`. `Transaction` rows are *never* rewritten — a pre-split BUY keeps its raw broker-reported qty/price forever. At read time, `RealizedGainCalculator.AvcoState.applySplit(factor)` multiplies `totalShares` in place and leaves `totalCost` unchanged (average cost divides by the factor). The walk consumes a per-security ascending `List<CorporateActionSplit>` through `applyPendingSplits(state, splits, cursor, txnDate)`, which advances a monotonic cursor — every event with `exDate <= txDate` fires before the transaction's `state.step(t)`. Both ledger-walk consumers — `HistoryQueryService.compute()` for the History page and `StockGroupService.computeTotalRealizedGain` for the overview card — batch-load splits for every in-scope master via `CorporateActionSplitRepository.findBySecurityMasterIdInOrderByExDateAsc(Set<UUID>)` and thread the resulting `Map<UUID, List<CorporateActionSplit>>` into the per-group AVCO state.

**Rationale:** Mutating the ledger destroys the audit trail — the whole point of `transactions` being the immutable source of truth (SYSTEM_SNAPSHOT §4). Worse, if the split feed is wrong (EODHD publishes a provisional ratio, a stock dividend is later reclassified), correcting a ledger-mutation approach means re-running a migration across every affected user's rows. The read-time path confines the fix to a single `corporate_action_split` row: delete it, update it, or add a manual override, and the *next* read produces the corrected P/L. It also folds stock dividends into the same mechanism for free — `factor = 21/20` is arithmetically identical to `factor = 20/1`, and the AVCO walk can't tell them apart. The batch-load + in-memory `Map` keeps the hot path at one extra SQL query per request (typically 0–3 rows per security), not N queries.

**Ex-date convention:** A transaction dated on or after `ex_date` is already post-split — this matches how every broker books transactions (the quote on ex-date morning is the post-split quote). The walk consumes a split when it encounters the first transaction with `txDate >= exDate`: if state is empty, the multiply is a safe no-op; if state is accumulated from earlier pre-split BUYs, the multiply scales it forward correctly. This removes any ambiguity about which transactions sit on which side of the event.

**Enforcement:** `RealizedGainCalculatorTest` covers 20:1 forward (AMZN case), 1:10 reverse (shares scale down, cost unchanged), `21:20` fractional stock dividend, the exact-ex-date boundary, multiple partial sells that straddle a split, and both the native and reporting-currency overloads. `HistoryQueryIntegrationTest.summaryAppliesStockSplitsToRealizedPnl` wires the full path against a real Postgres container — seeding a split through `CorporateActionSplitRepository`, persisting pre/post BUY+SELL transactions, and asserting that realized flips from phantom loss (≈ −$5,130 without split awareness) to the small profit the cash flows actually reflect.

## Decision 2: Canonical `security_master_id` Is the Only Join Key

**Choice:** Split rows reference `security_master_id` resolved to its canonical id (`COALESCE(canonical_master_id, id)`) at write time by `CorporateActionSplitBackfillJob.resolveEodhdProviderSymbol` and `recordManualSplit`. Consumers resolve transactions to the same canonical master id (`SecurityMaster.getEffectiveId()`) before grouping, so a holding that spans aliased listings — or moves between BSE/NSE — shares one AVCO state and one set of applicable splits.

**Rationale:** This is the same aggregation principle as ADR-012: `security_master_id` is the system's single source of truth for "what is this asset?" If splits were keyed by listing or market-data symbol, a security dual-listed across exchanges would require duplicate split rows and the rollup would have to de-duplicate at read time — a second place where a future listing alias could silently break the math. Keying by canonical master means the `MarketDataSymbolRepository.findActiveEodhdSymbolsWithMaster` projection collapses the listing layer at query time and feeds one `(providerSymbol, effectiveMasterId)` pair per canonical master.

## Decision 3: Three Ingestion Triggers, Idempotent, Constraint-Backed

**Choice:** `CorporateActionSplitBackfillJob` populates the table via three paths: (1) **JIT** via `@EventListener(HistoricalPriceBackfillCompletedEvent)`, so every newly added security picks up its lifetime split history as soon as price history lands; (2) **weekly sweep** at `0 0 4 ? * SUN UTC`, to catch forward-looking splits on securities already in the system; (3) **admin** — `CorporateActionAdminController` exposes `/api/admin/corporate-actions/splits/{backfill-all, backfill/{id}, manual}`, superuser-gated via the class-level `@PreAuthorize` pattern from `AdminController`, with a `requireSuperuser` defensive recheck. All three converge on `backfillForMasterWithProviderSymbol(canonicalMasterId, providerSymbol)`, which batch-loads existing `ex_date` values once per master, filters EODHD's response against that set in memory, and upserts the rest. Both `recordManualSplit` and the batch upsert catch `DataIntegrityViolationException` to tolerate the unique-constraint race, then reload the winning row.

**Rationale:** JIT + weekly is the standard market-data pattern in this codebase (same shape as historical-price and FX backfills) — it means the splits table is populated without anyone ever having to click an admin button during normal operation. The admin endpoints are the escape hatch for the three things that go wrong with any provider feed: the JIT event didn't fire, the provider missed a ratio, or the ratio is wrong. Reading existing `ex_date` values in one query instead of one-per-split kills an N+1 pattern that would otherwise grow with provider split density. Catching the unique-constraint violation rather than pre-checking eliminates the read-then-insert race window that concurrent triggers would otherwise expose — the constraint is the authority, not the application.

**No outer `@Transactional` on the catch-and-reload paths:** `recordManualSplit` and `backfillForMasterWithProviderSymbol` both run without a method-level `@Transactional`. This is deliberate: on PostgreSQL a constraint violation puts the current transaction into **aborted** state, and any subsequent SQL on that connection fails with `"current transaction is aborted, commands ignored until end of transaction block"`. Wrapping the method in `@Transactional` would therefore make the reload inside the `catch` block throw instead of returning the winning row — defeating the entire point of the catch. Relying on Spring Data's per-repository auto-transactions means the failing `save()` rolls back cleanly and the reload runs on a fresh transaction. Any future refactor that adds `@Transactional` here must first replace the catch-and-reload with a native `INSERT … ON CONFLICT DO NOTHING RETURNING *` or an equivalent savepoint strategy.

## Consequences

- `transactions` stays a perfect audit trail. No ledger row will ever carry a qty or price different from what the broker actually reported — an auditor, tax software, or a user cross-checking a Trading212 statement sees the exact same numbers the system was given.
- Correcting a bad split or a missed stock dividend is an UPDATE / INSERT on `corporate_action_split` — zero migrations, zero data backfill jobs, zero chance of partial recompute. The next read produces the corrected P/L for every affected user.
- The same mechanism handles stock dividends (fractional ratios) without any branch in the AVCO engine — they are arithmetically identical to integer splits from `AvcoState.applySplit(factor)`'s perspective. The `factor = ratioNumerator / ratioDenominator` form is the unifying abstraction.
- `RealizedGainCalculator` remains the single AVCO source of truth (ADR-012 principle); both the detail view and the aggregate rollup see identical realized numbers post-split, because both walk the same state machine with the same split list. Drift between surfaces is still architecturally impossible.
- Read cost grows by one batched SQL query per request (`findBySecurityMasterIdInOrderByExDateAsc(Set<UUID>)`), not N. For a portfolio of a few hundred securities with an average of zero-to-three splits each, this is sub-millisecond.
- Aliased cross-exchange holdings share one set of splits automatically because both transactions and splits resolve to the canonical `security_master_id` — no de-duplication logic required at the consumer.
- Fractional rounding is handled organically: `BigDecimal.divide` uses `MathContext.DECIMAL128` inside `factor()`, and the walk never re-computes a running total, so rounding error can't accumulate across sells. The broker's raw qty/price for the post-split SELL *is* the correct number; AVCO only has to scale the pre-split state to meet it.
- Phase-2 work (dividends as income, mergers/spin-offs) slots into the same table shape — a new `event_type` column and a new `AvcoState` handler leave the existing ingestion and consumer paths unchanged. The API contract is stable.
