# ADR-019: Read-Time Canonicalization for Duplicate `SecurityMaster` Rows

**Status:** Accepted
**Date:** 2026-04-23
**Context:** Ingestion pipelines — SnapTrade sync, manual add-transaction, Zerodha CSV — intermittently produced two `security_master` rows for the same underlying security when the listing-level dedup in `SecurityListingService.createListing(...)` missed (no `listing_figi`, no existing `(exchange, ticker)` listing). Nine clusters in Ekta Zerodha (EMBASSY, JIOFIN, MINDSPACE, MTARTECH, NESTLEIND, SBICARD, UJJIVANSFB, ARE&M, DREAMFOLKS) ended up with BUYs on one master and SELLs on another. The AVCO engine groups by `effective_master_id`, so the SELL-side master saw $0 cost basis and realized gain inflated to $8,724.49 on the Ekta Zerodha dashboard card. A structural answer was needed — not a one-off SQL repair — because the same ingestion gap will keep producing duplicates for any ticker that lands without a FIGI.

---

## Decision 1: Ingestion Dedup Is Best-Effort, Never-Fail

**Choice:** `SecurityListingService.createListing(...)` now performs a deterministic lookup **before** calling `securityMasterRepository.save(...)`. Priority 1 is `(exchange, ticker)` on `security_listing`; priority 2 is same-country across any exchange, accepting the match only when the candidate listings share exactly one distinct master (ambiguity → fall through). Any lookup error or ambiguity **falls through to create-new** — the service never throws from the dedup path. Log hits as `INFO`, ambiguity as `WARN`, so reuse-rate vs. ambiguity-rate becomes observable from production logs.

**Crucially, we do not add a DB-level unique constraint on `(ticker, country)` or any composite natural key on `security_master`.** SnapTrade ingestion is asynchronous and batched; a constraint violation from inside `createListing(...)` would roll back the caller's transaction and silently drop a broker-reported row. An over-eager duplicate is recoverable — link via `canonical_master_id` and the read path heals. A dropped transaction is not recoverable. The Never-Drop-Data rule wins; deduplication stays at the application layer with the existing `FigiResolutionService.applyResolvedResults()` reactive canonicalizer as the async safety net.

**Rationale:** The cheap alternative — add the unique constraint and catch `DataIntegrityViolationException` inside the service — was rejected because the catch point sits inside a method that may have already persisted a transaction row in the same Hibernate session; the rollback boundary isn't the master-save, it's the enclosing `@Transactional`. Recovering from a partial-rollback state across every ingestion call site (SnapTrade, CSV, manual entry) multiplies the failure surface. Best-effort plus reactive FIGI linking keeps the failure mode at "one duplicate row" instead of "one lost broker transaction."

## Decision 2: Pre-Existing Duplicates Heal at Read Time via `canonical_master_id`

**Choice:** Duplicates already in the database are collapsed by setting `security_master.canonical_master_id` on each duplicate to point at a chosen canonical target. **No transaction or holding row is ever modified.** Every read-side service — `PortfolioReplayService`, `RealizedGainCalculator`, `PortfolioSummaryService`, `StockGroupService`, `HistoryQueryService` — resolves the effective master via `SecurityMaster.getEffectiveId()` (`canonicalMasterId != null ? canonicalMasterId : id`) or the equivalent `COALESCE(canonical_master_id, id)` in SQL. Updating the self-FK heals the AVCO math on the next read; no replay runs, no cache busts.

A new admin endpoint `POST /api/admin/securities/link` (superuser-only) exposes this primitive operationally: `{targetMasterId, duplicateMasterIds[]}` → `security_master.canonical_master_id = target` on each duplicate, plus a transitive re-pointing pass that redirects any master already pointing at one of the duplicates so `getEffectiveId()` stays single-hop everywhere. Idempotent, atomic, non-destructive. The nine Category-1 clusters in Ekta Zerodha were collapsed this way; the inflated $8,724.49 figure resolved on the next dashboard fetch without any replay or ledger mutation.

**Rationale:** The immediately obvious alternative — re-point the `transactions.security_listing_id` or `holdings.security_master_id` FKs from the duplicate master to the canonical — was rejected because it breaches the Immutable Ledger principle (SYSTEM_SNAPSHOT §4). The ledger is truth: every row reflects what the broker reported, and we never back-write. Re-FK'ing a thousand transaction rows because two `security_master` rows exist is a ledger rewrite with no audit trail, no reversibility, and a bug surface (FK correctness across the entire replay graph) that no test catches structurally. Read-time canonicalization sidesteps the whole question: the `transactions` table stays byte-identical to what it was before the fix, and the same `effective_master_id` grouping that ADR-005 established for symbol-changes now carries duplicate-collapse too — one mechanism, two use cases, zero new invariants.

**The mechanism is the same primitive `FigiResolutionService` already uses.** When OpenFIGI resolves two masters to the same composite FIGI, the jobs module sets `canonical_master_id` on the loser — reactively, asynchronously, no ledger touch. The admin endpoint is the manual counterpart: it does exactly what the reactive path does, keyed off operator judgement instead of FIGI identity, for clusters where FIGI resolution is unreliable (Indian NSE/BSE tickers, most commonly).

## Consequences

- **`transactions`, `holdings`, `security_listing` are never mutated to fix master duplicates.** Ever. The immutable ledger principle holds at the asset-identity layer too. Future master-consolidation work adds canonical edges; it never rewrites FKs.
- **Read-path services must route through `getEffectiveId()` / `COALESCE(canonical_master_id, id)`.** This is already true of every aggregator on the hot path (see SYSTEM_SNAPSHOT §4 "Aggregation Key is Always `security_master_id`"); ADR-019 extends the discipline to any new surface. A service that queries by raw `security_master.id` without the coalesce is a latent bug.
- **Duplicate detection stays manual for now.** The admin endpoint takes explicit IDs; there is no "find all duplicate clusters for user X" auto-discovery. Adding that is a strict add-on — same endpoint, new finder service — not an architectural shift.
- **Category-2 orphans are outside scope.** Tickers with SELLs but zero BUYs anywhere in the ledger (ADANIPORTS, CESC, EXIDEIND, HINDZINC, ITCHOTELS, KOTAKBANK, NMDC, TATASTEEL, ULTRACEMCO in Ekta Zerodha) are a missing-data problem, not a master-identity problem. The ADR-019 mechanism doesn't and can't address them; they require ledger backfill or `excluded_from_calculations=true`.
- **Reactive and manual canonicalization are interchangeable at the data layer.** `canonical_master_id` doesn't record who set it (FIGI matcher vs. admin operator). That's fine — the read path only cares about the current value. If auditability of canonicalization events becomes a requirement, a `security_master_canonical_history` event table is the standard add-on; the current column stays the source of truth for reads.
- **The ingestion preventive layer and the read-time heal are independent.** The preventive path stops new duplicates; the read-time heal collapses existing ones. Neither depends on the other. Removing one does not force a rethink of the other.
