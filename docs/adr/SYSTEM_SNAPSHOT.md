# SYSTEM SNAPSHOT

High-density context for AI coding agents. Reflects the active state of the system. For deeper rationale, see the numbered ADRs in this directory.

---

## 1. Product Vision & Domain

An **institutional-grade personal portfolio management and wealth analytics platform**. It competes with high-end tools like Bloomberg Terminal and InvestingPro — **not** consumer apps (Robinhood, Mint). Priorities, in order:

1. **Accuracy** — every number the UI shows must be mathematically defensible.
2. **Rigorous financial math** — `BigDecimal` everywhere, historical FX preserved, AVCO as the single realized-gain algorithm, immutable transaction ledger as truth.
3. **Premium data density** — multi-currency, multi-broker, cross-exchange aggregation surfaced without horizontal bloat.

Users are sophisticated investors tracking global portfolios across brokers, currencies, and asset classes. "Correctness > Convenience" is non-negotiable.

## 2. Core Business Entities

- **User** — owner of everything; reporting currency is a user-level preference.
- **Portfolio** — a broker account (SnapTrade-synced) or a manual container; has dynamic transaction-backed holdings.
- **SnapTradeConnection / SnapTradeUser** — broker linkage; state = (`status`, `sync_phase`, `failure_reason`); async event-driven sync with polling. `failure_reason ∈ { UNSUPPORTED_BROKER, UNKNOWN }` is the structured signal the frontend reads to render category-specific copy ("Broker not supported …") instead of pattern-matching free-text `last_sync_error` (ADR-030).
- **Broker (broker_metadata)** — vetted-broker catalog. Active rows with non-null `snaptrade_slug` are the only brokers the UI can offer; `snaptrade_slug` matches SnapTrade's `referenceData.listAllBrokerages` catalog verbatim (uppercase, hyphenated — `INTERACTIVE-BROKERS-FLEX`, not `interactive_brokers`). Manual placeholder (`id=99`, slug NULL) is excluded from `GET /api/brokers`. Redis-backed `brokers` cache is cleared on every JVM start by `BrokerCacheStartupClearRunner` so deploy → migrate → restart is self-healing for slug fixes (ADR-030).
- **Transaction** — immutable ledger row; carries `raw_data` JSONB audit trail, `broker_fx_rate`, USD-normalized fields, and cached `normalized_reporting_*` projections. `TransactionType` = `BUY | SELL | DIVIDEND | TRANSFER | BONUS | SPLIT | MERGER_IN | MERGER_OUT | SPINOFF_IN | RIGHTS_IN` — the last six are per-account broker emissions retained for audit only; **`PortfolioReplayService` filters them out of the AVCO timeline** so the EODHD/BhavKosh `corporate_action_*` row is the sole authority for position math (ADR-029). CSV import refuses to write them at ingest (`RowStatus.CORPORATE_ACTION_DROPPED`); SnapTrade still writes them but they are inert under the replay filter.
- **Holding** — current position; references `security_master_id` (canonical), `acquired_listing_id`, nullable `override_market_data_symbol_id`, and `broker_raw_ticker` (never overwritten). `HoldingDto` carries `securityMasterId` (effective canonical master via `getEffectiveId()`) for dismiss-review and `reviewState` + `reviewActionHint` (including `suggestedTransactionDate`) for triage CTAs.
- **SecurityMaster / SecurityListing / MarketDataSymbol** — three-layer identity: *what it is* → *where it trades* → *how it's priced*. `canonical_master_id` self-FK handles deduplication. `reviewDismissed` boolean (DEFAULT FALSE) suppresses the Needs-Review shelf badge without altering any calculation (ADR-032); distinct from `Holding.excludedFromCalculations` which removes the position from all totals. `market_data_symbol.symbol_kind` (`SymbolKind` enum — `EQUITY | RIGHTS_ENTITLEMENT | REIT | BOND | ETF | UNKNOWN`) is the structured-classification surface read cross-language; populated by `SymbolKindClassifier` at every write site so the price-sweeper can skip categories (e.g. `RIGHTS_ENTITLEMENT`) via a column filter, never a SQL string LIKE (ADR-040). `market_data_symbol.backfill_status` (`PENDING | SUCCEEDED | FAILED | NO_DATA`, NULL when never attempted) is the responsive terminal verdict written by `EodhdHistoricalPriceBackfillJob` on every retry-exhaustion / non-retryable / empty-bars-no-prior-data exit — read by `PortfolioBackfillStatusService` to drop listings from `pendingListingIds` within one poll cycle instead of waiting on the 15-minute staleness fallback (ADR-041).
- **ManualAsset** — user-level static assets (cash, gold, real estate). **Owned by User, never by Portfolio.** Supports `monthly_increment` via a scheduled physical-apply job.
- **UserAssetExposures** — pre-computed per-user per-asset-class read model for the allocation widget.
- **UserAssetClassTargets** — policy-level allocation rules (target %, max drift %).
- **MarketDataPriceDaily / LatestMarketPrice / FxRate** — historical OHLCV (EODHD/BhavKosh), latest prices fed by the 15-min `LatestPriceRefreshJob` (EODHD bulk for non-Indian, Zerodha individual for Indian) and the Python sweeper sidecar (Fyers `/quotes`, Indian-only, sidecar-side gated by market window), plus an async EODHD JIT on cold cache+DB miss for non-Indian listings; historical FX (EODHD). The sidecar resolves provider symbols via the Fyers public symbol master (downloaded daily, indexed `by_isin` then `by_ticker`, walks `COALESCE(canonical_master_id, id)` to read the canonical ISIN) — no hardcoded `(prefix, series)` synthesis (ADR-040). Reads serve from the `latestPrices` Redis region (24h backstop, `@CachePut`-driven) or the `latest_market_price` DB row — never a synchronous provider call on a user thread. The dropped `is_realtime` column had zero readers; freshness derives from `as_of` (ADR-038).
- **CashFlow** — non-trade portfolio cash movement (`flow_type` ∈ `DEPOSIT | WITHDRAWAL | FEE | INTEREST | TAX | FX_CONVERSION | ADJUSTMENT`), keyed by `portfolio_id`, signed `amount` stored verbatim from the broker payload. Populated by `SnapTradeActivityMapper` for SnapTrade activities with no security listing **and by `CsvImportService` at commit** for CSV rows resolved to `RowStatus.CASH_FLOW`; both paths collapse the noisier broker vocabulary (`CONTRIBUTION`/`MANAGEMENT_FEE`/`WITHHOLDING`/`CONVERSION`/`FX`) into the canonical seven. CSV uses a separate `csv:cf:v1:` external-id hash and a within-batch dedup pass to short-circuit collisions on the partial unique index. **Write-only from analytics today** — exists for audit completeness and future cash-balance reconciliation. Synthetic external_ids stay on `broker_activity_log` only and never propagate to `cash_flows.external_id` (ADR-028).
- **BrokerActivityLog** — immutable raw-payload audit row for every SnapTrade `Activity`; `parsed_status` ∈ `PENDING | PARSED | UNMAPPED_TYPE | …` and `parsed_entity_type` ∈ `TRANSACTION | CASH_FLOW | CORPORATE_ACTION` is the polymorphic discriminator routing each row to its destination table. `POST /api/v1/admin/snaptrade/replay-unmapped` (superuser-gated, keyset-paginated, idempotent) re-runs the parser against UNMAPPED rows when parser coverage extends (ADR-028).
- **SecurityListingRenameHistory** — append-only audit of ticker changes detected against a same-ISIN anchor, unique on `(canonical_master_id, old_ticker, new_ticker, exchange_code)`. Sole writer today: `SecurityListingRenameDetectionJob` (daily, Fyers public-master diff for NSE/BSE, `source = 'FYERS_MASTER_DIFF'`, gated by `app.jobs.rename-detection.enabled=false` until canonical-master ISIN coverage clears 80% — currently 0%). Pattern: detect → mint alias via `SecurityListingService.resolveOrMintMaster` (ADR-034 seam) → create new listing under alias → write audit row → publish `MasterAliasLinkedEvent`. Never participates in resolution — read-side canonicalization still flows through `security_master.canonical_master_id`. Pre-check by the unique tuple short-circuits sequential re-runs; the DB constraint is the race safety net. `ExchangeSymbolLoadJob` is queued to join as a second writer (`source = 'EXCHANGE_LOAD_JOB'`) once its in-place `setTicker` mutation is removed — that mutation is the open ADR-019 violation noted in ADR-040 §Consequences (ADR-019, ADR-040).
- **CorporateActionSplit** — keyed by canonical `security_master_id`; read-time multiplier (`factor = ratioNumerator / ratioDenominator`) applied in `RealizedGainCalculator.AvcoState.applySplit` during the AVCO walk. Covers stock splits (20:1, 1:10 reverse) and stock dividends (21:20) identically. Ingested by `CorporateActionSplitBackfillJob` via JIT event, weekly sweep, and superuser admin endpoints (ADR-014).
- **CorporateActionCashDividend** — declared cash-dividend events (EODHD `/api/div/{SYMBOL}`), keyed by canonical `security_master_id`, `(security_master_id, ex_date)` unique. **Reconciliation-only, never read by analytics.** All user-visible dividend totals (`HoldingDto.totalDividends*`, dashboard per-portfolio `totalDividendIncome{Lifetime,Ytd}`) derive exclusively from `transactions` rows of type `DIVIDEND` via `DividendAnalyticsService`, summing `normalized_reporting_amount` at historical FX. The events table feeds `DividendAnomalyService` only — an admin-gated `/anomalies` endpoint with a `minExpectedTotal` triage filter — to flag broker-feed gaps against company declarations. Ingested via JIT event, Sunday 04:30 UTC sweep, and superuser admin endpoints (ADR-017).
- **CorporateActionMerger / CorporateActionSpinoff** — declared cross-master events that drive basis transfer between `security_master_id`s at an exact ex_date UTC. Merger: `(from_master_id, to_master_id, ex_date, ratio_numerator/denominator, cash_per_share?, cash_basis_pct_override?)`, unique on `(from_master_id, ex_date)`. Spin-off: `(parent_master_id, child_master_id, ex_date, shares_per_parent, basis_allocation_pct)` — pct stays **nullable**; authoritative source is IRS Form 8937 entered via the superuser admin endpoint. Replay treats null pct as `SPINOFF_MISSING_BASIS` (skip transfer, parent retains 100% basis). Consumed exclusively by `PortfolioReplayService` at read time via `AvcoState.mergeFrom / applyCashBoot / splitOffFraction`; emit ephemeral `CorporateActionLedgerEntry` rows that surface on the History page as `synthetic=true` audit rows (never persisted). Merger ingest: EODHD `/api/fundamentals` is the only auto-path on `CorporateActionMergerBackfillJob`, and it is **gated off by default** (`app.jobs.corporate-actions-mergers.enabled=false`) — the production subscription tier returns HTTP 403. Manual entry via `recordManualMerger` is unaffected. Spin-off ingest: **manual entry only** via `CorporateActionSpinoffService.recordManualSpinoff`; no JIT listener, no weekly sweep, no auto-backfill on any provider (ADR-018, ADR-024, ADR-027).
- **FyersToken** — singleton row holding the system-wide Fyers V3 daily access token. Fixed `id = '00000000-0000-0000-0000-000000000001'`; `access_token` and `refresh_token` are AES-256-GCM ciphertext via `EncryptedStringConverter`; `repository.save()` upserts by primary key, so `POST /api/v1/admin/fyers/exchange` (superuser-only) overwrites without delete. `count(*)` is always 0 or 1. The pattern generalises: any future system-identity vendor (Kite, Upstox, …) gets its own one-row table — never reuse `SnaptradeUser`-style per-user storage for credentials shared across all callers (ADR-036).
- **FeatureFlag** — per-environment runtime toggle row keyed on `flag_key VARCHAR(128)` PK; `is_enabled BOOLEAN` plus `activated_at TIMESTAMPTZ` set only on the false→true edge (Days Active = `now() - activated_at` is the retirement signal). Each environment writes to its own DB so flag state is naturally scoped without schema gymnastics. `FeatureFlagService.isEnabled(key)` is `@Cacheable("featureFlags", key="#flagKey")` and **fail-closed** — unknown keys, cache misses, and DB/Redis outages return `false`. Admin mutations at `/api/v1/admin/features` (superuser-gated) carry `@CacheEvict(value="featureFlags", allEntries=true)`; public `GET /api/v1/features` returns only enabled flag keys (disabled keys often name in-flight features and must not leak). Frontend `FeatureFlagService` (Angular) hydrates a `Set<string>` signal on app boot and after every admin toggle (ADR-037).
- **Queues** — `figi_resolution_queue`, `ticker_resolution_queue`, `global_ticker_mapping` (crowdsourced cache).
- **Copilot Chat Sessions** — LLM-assisted analysis surface (currency-normalized inputs required).

## 4. Strict Architectural Principles

- **Accounting Tie-Out Principle** — every row must sum to the summary header. Holdings rows and group summaries compute through the same `ValuationEngineService`; realized gain flows through a single `AvcoState` state-machine (`RealizedGainCalculator`).
- **History / Ledger View — Server-Authoritative, URL-Driven** — `/api/history/{summary,transactions,closed-positions,missing-transactions}` (ADR-013) replays the ledger once per request through the same `AvcoState` to emit raw txns, closed-episode rows (qty transition `>0 → 0`, with `episodeIndex` for re-opens), and ledger-only anomaly rows (`SELL_BEFORE_BUY`, `NEGATIVE_QTY`). Filter/tab/page state lives in URL query params; broker-sync gap detection deferred until persistent broker-position snapshots exist.
- **Unified Valuation Engine** — all reporting-currency cost basis, gain/loss, and FX impact math lives in one place. Any new aggregator calls in; never re-implements.
- **Immutable Ledger, Cached Projection** — `transactions` is truth and is never rewritten. Corporate actions (stock splits, stock dividends, mergers, spin-offs) live in `corporate_action_*` tables and are applied **at read time** inside `AvcoState` during the ledger replay — never back-adjusted into transaction rows (ADR-014, ADR-024). `normalized_reporting_*` columns are a rebuildable cache. Changing reporting currency or correcting a split invalidates the cache, never the ledger.
- **`corporate_action_*` Is the Only AVCO-Authoritative Source for Corporate Events** — broker-reported `Transaction` rows of types `BONUS, SPLIT, MERGER_IN, MERGER_OUT, SPINOFF_IN, RIGHTS_IN` are filtered out of `PortfolioReplayService`'s timeline (`BROKER_REPORTED_CORPORATE_ACTION_TYPES`). They remain in the ledger for audit but contribute zero to position math; the EODHD/BhavKosh-populated `corporate_action_*` tables are the sole moving force. CSV import refuses these rows at ingest (`RowStatus.CORPORATE_ACTION_DROPPED`); SnapTrade still writes them but they are inert under the replay filter. `HoldingService.persistHoldingFromReplay` reads `transaction.securityListing.security` directly — the resolver is not consulted at read time, eliminating the `AmbiguousIdentifierException` WARN cascade for ambiguous tickers (ADR-029). **User-initiated rights/free allotments must use `transactionType='BUY'` with `price=0`** — never `RIGHTS_IN`, which is silently dropped. BUY @ price=0 enters the AVCO walk correctly (cost basis=0, full proceeds become realized gain) and is the canonical form for this case (ADR-032).
- **Master Deduplication via Read-Time Canonicalization** — never modify transaction or holding rows to fix duplicate `security_master` entries. Set `security_master.canonical_master_id` on the loser; every aggregator resolves via `SecurityMaster.getEffectiveId()` / `COALESCE(canonical_master_id, id)` at read time. Ingestion dedup is best-effort (never throws), and the superuser `POST /api/admin/securities/link` endpoint is the manual counterpart to the async FIGI canonicalizer. **Ticker renames extend the same rule**: `SecurityListingRenameDetectionJob` mints a new listing under the canonical master via `SecurityListingService.resolveOrMintMaster(newTicker, ..., isin)` — never updates `security_listing.ticker` or flips `is_active` on the old row — and writes an audit row to `security_listing_rename_history`. Both old and new listings remain active under the same canonical master; CSV dedup hash idempotency (`CsvImportService.computeRowHash`) is preserved by construction (ADR-019, ADR-040).
- **Null-Exchange Listing Promotion at Write Time** — `SecurityListingService.findOrCreateByTickerOrFigiOrExchange` Step B' promotes an existing `exchange_id IS NULL` sibling rather than creating a duplicate when an exchange resolves late, gated on currency match + zero transactions on the candidate + exactly one match. Promotion re-fires `JitSecuritySetupService.onNewListingCreated` and re-enqueues FIGI because the original Phase α JIT bailed for the null-exchange row. Backed by partial index `idx_security_listing_ticker_null_exchange`. Composes with ADR-019: promotion eliminates the listing-level duplicate at write time; canonicalization is the master-level safety net when promotion's gates don't fire (ADR-028).
- **EODHD JIT Backfill Is Serial, Throttled, Bounded-Retry** — `EodhdHistoricalPriceBackfillJob.onBackfillRequested` runs on the dedicated single-threaded `eodhdHistoricalBackfillExecutor` (queue 1000, `CallerRunsPolicy`), with a 200ms pre-call throttle and a 3-attempt linear-backoff retry (1s → 2s) on `EodhdApiException` (network I/O, 5xx, transient rate-limit). `HistoricalPriceBackfillCompletedEvent` is the load-bearing trigger for `CorporateActionSplitBackfillJob` — transient upstream failure must not silently break the splits cascade. The shared `backgroundJobExecutor` must not be reintroduced for this listener. Pattern generalises: per-provider serial executors are the right home for any external-API JIT consumer that fan-outs a fragile event chain (ADR-028).
- **Latest-Price Read Path: 15-Min Cache + Symmetric Async JIT, No Sync HTTP on User Threads** — `GET /api/stocks/{id}/current`, `/figi/{id}/current`, and `/batch/current` serve from the `latestPrices` Spring cache (24h backstop, `@CachePut`-driven by `LatestPricesRefreshedEvent`) or the `latest_market_price` DB row — never a synchronous Zerodha/EODHD call on a user thread. The legacy `stockPrices` 5-min cache is removed. Cold cache+DB miss enqueues an async JIT: Indian listings → sweeper `/force-fetch` (Fyers); non-Indian → `MarketDataService.enqueueEodhdJitFetch` on the dedicated single-threaded `eodhdJitExecutor`, handled by `EodhdJitFetchHandler` with the same 200ms throttle + 3-attempt linear-backoff retry pattern as ADR-028. Concurrent reads of a cold symbol dedupe through an in-process `ConcurrentHashMap<String, CompletableFuture<Void>>` keyed by namespaced symbol (`yahoo:` / `eodhd:`). `LatestPriceRefreshJob` (cron every 15 min) skips the EODHD partition when `MarketHours.isUsMarketOpen(now)` is false and the Zerodha partition when `MarketHours.isIndianMarketOpen(now)` is false (DST-aware via `ZoneId`); BHAVKOSH keeps running because the Python sweeper is already market-window-gated. Both the scheduled batch and the JIT path acquire the fair-FIFO `EodhdRateLimiter` bean so a JIT burst can't poison the next batch with a 429. `POST /api/stocks/refresh/{id}` now persists + emits the cache-update event (was previously a fetch-only call that silently served the `@Cacheable` stale value). Holiday calendars and sweeper-side market-window gating for `/force-fetch` are deferred (ADR-038).
- **Connected-Component Valuation Engine** — `PortfolioReplayService` walks each portfolio as a set of `ReplayChain`s (connected components of masters under merger + spin-off edges). Cross-master basis transfer fires at an exact ex_date UTC timestamp via `AvcoState.mergeFrom / applyCashBoot / splitOffFraction`; audit surfaces as ephemeral `CorporateActionLedgerEntry` rows on the History page (`synthetic=true`). The `transactions` table stays strictly append-only — no merger/spin-off row is ever written to it, and the `Transaction.TransactionType` enum is not extended. Single engine across `HistoryQueryService`, `HoldingService`, `StockGroupService`, `PortfolioSummaryService` — Holdings persistence reads `ReplayResult.terminalByMaster()` (no parallel AVCO loop). The shadow-mode flag and legacy paths were removed once parity burned in (ADR-018, ADR-024).
- **Session-Scoped View Toggles, Not Profile Writes** — global UI context modifiers (notably the reporting-currency dropdown) live in session-only Angular Signals (`NavbarStateService`). Two seed sources, both gated by a single `sessionOverridden` flag that flips true on any user dropdown click: `seedFromUserIfUntouched(profileCurrency)` (called from `AppComponent` on `currentUser$` emit and on every `NavigationEnd`, but **skipped on `/portfolio` routes**) and `seedFromPortfolioIfUntouched(portfolioCurrency)` (called from `PortfolioComponent.loadPortfolios` when exactly one portfolio is in context, with a fallback to the user-profile seed for multi-portfolio mode). The portfolio-route guard prevents an EUR-then-INR flicker for users whose profile currency differs from their broker portfolio's native currency. After seeding, `PortfolioComponent.reportingCurrency` is read back from the signal **synchronously** before `loadHoldingsForPortfolios` fires so the holdings HTTP request and the constructor-effect-driven `onReportingCurrencyChange` cannot race on different currencies. Seeds MUST NOT call `PUT /api/users` or write to `localStorage` — profile-level currency changes publish `ReportingCurrencyChangedEvent` and force a full `normalized_reporting_*` rebuild across the user's ledger, which is correct for a Settings action and catastrophic for a header toggle. Backend endpoints respect the signal via an optional `?reportingCurrency=XXX` param with a profile fallback (ADR-016).
- **Holdings Are Strictly Derived** — the `holdings` table is a rebuildable projection of the `transactions` ledger. No HTTP endpoint creates or deletes a `holdings` row directly; the only mutation surface is `PUT /api/holdings/{id}` for metadata fields (`buyBelowPrice`, `maxAllocationPct`, `excludedFromCalculations`, `note`). Insertion and deletion happen only through replay (`HoldingService.updateHoldingsForPortfolio` writing every `ReplayResult.terminalByMaster()` output; `HoldingService.updateHoldingForMaster` writing exactly one) and `deleteByPortfolioId` cascades on portfolio delete. No import pipeline writes here directly; all imports produce real transaction types (`BUY`, `SELL`, `DIVIDEND`, `TRANSFER`) and the replay engine derives the holding from those. Oversold qty is *not* clamped — it persists as `quantity = 0` and is surfaced via `ReviewState.OVERSOLD`. `POST /api/holdings/rebuild?portfolioId={id}` forces a projection rebuild without touching the ledger (ADR-020, ADR-024, ADR-031).
- **Corporate-Action Listeners Recompute One Master, Not the Whole Portfolio** — `HoldingsRecomputeOnCorporateActionListener` (and any future `@TransactionalEventListener` consuming `MasterCorporateActionsRefreshedEvent`) routes through `HoldingService.updateHoldingForMaster(portfolioId, masterId)` — never `updateHoldingsForPortfolio`. The per-master path still replays the full ledger (cost basis is FIFO/AVCO-dependent across the portfolio) but persists exactly one Holding row, so a single JIT-seeded split cannot cascade into a portfolio-wide rewrite that resurrects skeleton Holdings for every master in the ledger. The portfolio-wide method is reserved for callers that intentionally rebuild every position (CSV import commit, `POST /api/holdings/rebuild`). `persistHoldingFromReplay` backfills `Holding.broker_raw_ticker` from the acquired listing's ticker only when currently NULL — the "never overwritten" rule on broker-specific suffixes (e.g. `RELIANCE-EQ`) is preserved (ADR-039).
- **Server-Classified Holdings Triage** — `LedgerGapDetector` (in `core`) maps `ReplayResult` outputs to a `ReviewState` enum (`OK | OVERSOLD | ORPHANED_BUY_NEEDS_RENAME | MISSING_LOCAL_EVENT | MISSING_GLOBAL_RENAME`) plus a structured `ReviewActionHint` (includes `suggestedQuantity`, `suggestedTransactionDate` = day before earliest SELL). `HoldingDto` carries both fields plus `securityMasterId`; the frontend `@switch`-renders and never derives. Excluded holdings (`excludedFromCalculations = true`) and dismissed masters (`reviewDismissed = true`) are exempt from classification; dismissed masters suppress synthetic ghost generation entirely. OVERSOLD rows render three triage actions: "Add missing BUY" (opens `add-buy-dialog`), "Mark as rights allotment" (records BUY @ price=0), and "Dismiss" (sets `SecurityMaster.reviewDismissed`); `-RE` tickers auto-promote "Mark as rights allotment" to primary. `MISSING_GLOBAL_RENAME` is admin-resolvable only; users cannot write to `corporate_action_*` tables (ADR-024, ADR-032).
- **AI-Assisted Mapper Pattern** — LLM involvement in CSV import is limited to schema inference: header + 5 sample rows in, `MappingProposal` JSON out. Java parses the full file deterministically via Apache Commons CSV using the user-confirmed mapping. The LLM never touches numeric values, dates, or the commit path. Every non-null column name in the proposal is validated against the real header server-side before use (injection defence). One LLM call per import regardless of file size. The mapping schema extends to heterogeneous broker formats (Freetrade, Trading 212, …) via four optional `ColumnMapping` fields — `directionColumn`, `dividendPriceColumn`, `skipTypes`, `typeAliases` — that the LLM populates and the user never edits (ghost state). `RowStatus` resolves to one of `NEW | DUPLICATE | SKIPPED | INVALID | CASH_FLOW | CORPORATE_ACTION_DROPPED`: cash events route to `cash_flows`, broker corp-action labels are dropped at ingest (canonical source is `corporate_action_*` per ADR-029), genuinely-unrecognised rows stay `SKIPPED` for visibility, and the preview surfaces a "show skipped categories" expander so the user can reclassify a category back to a trade type and re-run preview. `AiCsvMapper` pre-seeds corporate-action broker labels into the LLM-supplied `skipTypes` so they appear auto-handled from the first preview round-trip (ADR-020, ADR-026, ADR-029).
- **CSV Dedup is Strict — No Ordinal Fallback (`csv:v2:`)** — the row dedup hash (`externalId`) is keyed on `portfolioId | ticker | isoTimestamp(UTC) | type | qty | price | currency | externalTradeId`. Two rows that hash identically are duplicates; no row-ordinal suffix is ever appended to invent uniqueness. Partial fills from broker tradebooks (e.g. Zerodha `trade_id`) require the user to explicitly map the `externalTradeIdColumn` — silent position inflation via fabricated uniqueness violates the Accuracy principle. Re-upload of any file is always idempotent. Timestamp parsing treats naive `LocalDateTime` strings as UTC verbatim (no regional zone conversion), so the trade date never shifts across the midnight boundary (ADR-021).
- **Bare-CSV Listing Resolution — Broker Alias, FIGI Writeback, Manual Override** — broker-supplied exchange descriptions ("London Stock Exchange", "Knight Equity Markets LP") and broker-form tickers (Trading 212's `BMWD` → canonical `BMW3`) are normalised at ingest by `BrokerExchangeAliasResolver` / `BrokerTickerAliasResolver` (hardcoded `core` maps). Listings that still land with `exchange_id = NULL` (ticker+currency-only CSVs) hit OpenFIGI ticker-only; `FigiResolutionService` persists the resolved Bloomberg `exchCode` to `figi_resolution_stage.resolved_exch_code` and reverse-maps it via `BLOOMBERG_TO_EXCHANGE_CODE` to back-fill `listing.exchange_id`, then triggers `JitSecuritySetupService.onNewListingCreated` so EODHD price backfill can run. The `GR → XETRA` reverse-map tie-break is documented and known-imperfect for Berlin-only / Frankfurt-floor-only securities; the manual override endpoint `PATCH /api/figi-resolution/listing/{id}/exchange` (admin-token-gated) and the bulk repair endpoint `POST /api/figi-resolution/repair-unresolved` close the residual gaps. Ticker-only matches use a separate `TICKER_CONFIDENCE_THRESHOLD = 0.30` (ISIN-based stays at `0.90`) since validation already hard-rejects currency mismatches (ADR-027).
- **Backfill Progress is Hybrid — Derived for In-Progress, Persisted for Terminal** — async historical-price backfill jobs (`EodhdHistoricalPriceBackfillJob`, `CorporateActionSplitBackfillJob`) hold no progress state on a portfolio-level row. The CSV path's `GET /api/v1/csv-import/backfill-status` and the per-portfolio path's `GET /api/v1/portfolio-sync-status` derive `{status, syncPhase, completedCount, totalCount, pendingListingIds}` from `market_data_price_daily` over the user's owned listings, with three pending-set filters: FAILED `ticker_resolution_queue` entries (joined via `holdings.acquired_listing_id`), `market_data_symbol.backfill_status IN ('FAILED', 'NO_DATA')` — the per-symbol terminal verdict written by the JIT job on every retry-exhaustion / non-retryable / empty-bars-no-prior-data exit, clearing within one poll cycle (ADR-041) — and stale `MarketDataSymbol` rows whose `created_at` is older than **15 minutes**. The 15-min staleness window stays as a backstop for the path where the job dies before writing terminal status (process crash, executor queue overflow); it is no longer the primary clearing mechanism. The frontend renders at most one banner at a time with precedence SnapTrade > CSV > Prices; the SnapTrade "All data synced!" success copy is suppressed while prices are still SYNCING. The prices banner persists its poll start timestamp to `sessionStorage` so the 10-min stuck-state and 30-min hard-cap thresholds cannot be reset by reloading. No per-portfolio `*_backfill_status` table, no SSE/WS, no `BACKFILLING_PRICES` lifecycle on `SnaptradeConnection.sync_phase` (ADR-022, ADR-028, ADR-041).
- **Never Drop Data** — SnapTrade ingestion persists first (raw JSONB + `broker_raw_ticker`) and resolves asynchronously. Holdings are usable before pricing resolves.
- **Closed-Loop Broker Selection** — the frontend can never display, and the backend will never link to, a broker absent from `broker_metadata`. `GET /api/brokers` filters server-side to active rows with non-null `snaptrade_slug`; `POST /api/brokers/snaptrade/connect/start` accepts `brokerId: Integer` only, and `SnapTradeService.start` resolves the slug from `BrokerService.getBrokerById` before forwarding `.broker(slug)` to the SnapTrade SDK. Slugs are case- and punctuation-sensitive on the wire — the DB stores them in canonical SnapTrade form, sourced via the superuser endpoint `GET /api/v1/admin/snaptrade/brokerages` (wraps `referenceData.listAllBrokerages`). Failure routing is structured (`SnaptradeConnection.failure_reason` enum); the frontend `@switch`-renders on it and never pattern-matches `lastSyncError` text. `UnsupportedBrokerException` is the safety net for API tampering / deactivation races, not a user-visible flow. The Redis-backed `brokers` cache clears on every startup via `BrokerCacheStartupClearRunner` so any seed-fix migration is correct on next deploy without operator intervention (ADR-030).
- **SnapTrade Activity Routes to Three Destinations, Polymorphically** — every `broker_activity_log` row lands in exactly one of `transactions` (trades + per-account corp-action receipts; `parsed_entity_type='TRANSACTION'`), `cash_flows` (non-trade cash with no security listing; `parsed_entity_type='CASH_FLOW'`), or stays `UNMAPPED_TYPE` for operator triage. The `corporate_action_*` tables are populated by EODHD/BhavKosh jobs and admin endpoints only — SnapTrade Activity never writes to them. `MERGER` direction is resolved by sign-of-quantity; null/zero stays UNMAPPED. `OPTION_*` types stay UNMAPPED (no AVCO treatment yet). Adding a new activity type is a routing decision: trade-shaped with AVCO impact → `transactions`; pure cash → `cash_flows`; otherwise UNMAPPED. New `Transaction.TransactionType` values require corresponding `RealizedGainCalculator.applyTransaction` math + CSV alias coverage — otherwise prefer `cash_flows`. Synthetic external_ids never leave `broker_activity_log`. `SnapTradeActivityMapper` (extracted in D3) owns the switch (ADR-028).
- **System-Identity Vendor Tokens — Singleton Row, Superuser-Captured, Server-Side Exchange** — credentials that authenticate the application (not the user) live in a dedicated single-row table per vendor, keyed by a fixed UUID, with sensitive fields encrypted via `EncryptedStringConverter`. The OAuth-style code grant runs through a class-level `@PreAuthorize("hasRole('SUPERUSER')")` admin controller (`FyersAdminController` is the canonical instance) that exchanges the auth code via `RestClient` server-side using the secret stored only in `application.properties` — the frontend supplies only the public `client_id` and a runtime-derived `redirect_uri`, never the secret. Status endpoints return `{present, expiresAt, refreshedAt, expired}` and **never** include the token. The frontend `superuserGuard` is UX only; the `@PreAuthorize` is the security boundary. Admin surfaces sit under `/admin/...`, not `/settings/...` (different audience, different RBAC story) (ADR-036).
- **Feature Flagging — Guard at the Perimeter, Branch by Abstraction** — flags gate features at the **entry point only**. New endpoint / new route / new top-level UI surface: throw `ResponseStatusException(NOT_FOUND)` in the controller (backend) or short-circuit via a `CanActivate` route guard / single top-level `@if (featureFlags.isEnabled(...))` (frontend). Inline `if (enabled)` checks deep in services, calculators, or shared templates are forbidden — they make the legacy path uncleanable. Behavior-changing flags use Strategy + factory: extract the affected logic to an interface, keep the current code as `Legacy{Name}Impl`, add `V2{Name}Impl`, and choose between them in a `@Configuration` `@Bean` factory (backend) or a `useFactory` provider in `app.config.ts` (frontend) that consults `FeatureFlagService.isEnabled(...)` at injection time. Spring's `@ConditionalOnProperty` is not viable — flags are runtime-mutable in Postgres/Redis, not startup-time. **Fail closed by contract:** `FeatureFlagService.isEnabled(key)` returns `false` for unknown keys, cache misses, and DB/Redis outages — never override with permissive defaults. Flag state lives in the `feature_flag` table (per-env DB, naturally environment-scoped), Redis-cached under `featureFlags` (5-min TTL, evicted on every admin write). Admin UI at `/admin/feature-flags` is superuser-gated; the "Days Active" column is the retirement signal — a flag on in prod past its rollout window is tech debt and the abstraction must be collapsed (delete `Legacy*`, rename `V2*` to canonical, drop the row). Public `GET /api/v1/features` returns only the enabled-key list to avoid leaking disabled flag names (ADR-037).
- **Money = `BigDecimal`, Time = `Instant`/`OffsetDateTime` (UTC)** — never `double`/`float`/`Date`/`LocalDateTime`.
- **FX Fallback Hierarchy (in order):** broker FX → local historical FX (`fx_rate` table) → live FX snapshot. Missing rate degrades gracefully, never breaks the UI.
- **All-or-Nothing USD Normalization** — the USD-normalized cost basis path activates only when BUY transaction quantity ≥ held quantity. Otherwise fall back to live FX. No blending.
- **No Sync External Calls on Hot Path** — EODHD/OpenFIGI/Yahoo only in scheduled jobs, admin endpoints, or sidecars. Reads serve from DB/Redis.
- **Chunked Async Backfills** — event-driven (`@TransactionalEventListener(AFTER_COMMIT) @Async`), batched (100 rows), DB-only inside the loop.
- **Postgres-as-Queue** — `SELECT FOR UPDATE SKIP LOCKED` for FIGI/ticker resolution pollers; no Kafka/RabbitMQ.
- **Module Boundaries Enforced** — `core` = domain only; `api` = HTTP/security; `jobs` = headless scheduled work. `jobs` never imports `api`.
- **LLM Ingestion** — normalize currencies (to reporting currency) before any LLM input.
- **Aggregation Key is Always `security_master_id`** — never listing or symbol; cross-exchange holdings must consolidate.
- **Sidecar Is a Read-Only Consumer; Domain Classification Lives in Java** — the Python `price-sweeper` (and any future cross-language reader) consumes structured columns on `market_data_symbol` / `security_master` / `security_listing` and never owns broker-specific string heuristics. Skipping `RIGHTS_ENTITLEMENT` rows is a `symbol_kind <> 'RIGHTS_ENTITLEMENT'` filter, not `WHERE ticker NOT LIKE '%-RE'`. `SymbolKindClassifier` (in `core/util`) is the single classifier wired into every `market_data_symbol` write site — `BhavKoshSymbolDerivationService`, `SnapTradeSymbolExtractor`, `EodhdSymbolMappingJob`, `ZerodhaSymbolMappingJob`, `PriceService` JIT, `HistoricalPricePrefillJob`, `TickerMappingApplier`. `IndianTickerNormalizer` strips broker series suffixes (`-BZ/-BE/-T/-...`) at SnapTrade ingest, scoped to NSE/BSE only with `-RE` deliberately preserved (ADR-032 dependency); raw broker ticker survives on `holdings.broker_raw_ticker`. Fyers public-CDN dependencies (`FyersSymbolResolver` in Python, `FyersSymbolMasterClient` in Java) share a fail-soft contract — degraded mode on download failure, never raise on startup, `/health` exposes `fyers_resolver_available` (ADR-040).
- **Fyers `/quotes` Runtime Failure Classification — Tri-State, Negative Cache Never Poisoned by Transients** — `fetch_bulk_prices` returns `BulkFetchResult(prices, rejected)` and `fetch_single_price` returns `SingleFetchResult(outcome, data)` with `FetchOutcome ∈ {FOUND, REJECTED, FAILED}`. A symbol is **REJECTED** only when Fyers gave a successful top-level response and explicitly rejected the entry (per-entry `s == "error"`, missing `n`/`lp`, non-positive price, or unmappable Yahoo suffix). Network exceptions, top-level `s == "error"`, and token-error codes `{-15, -16, -17, -300}` are **FAILED** — the caller leaves those symbols out of *both* sets so the negative cache cannot be poisoned by a token expiry or transport blip. REJECTED symbols are negative-cached at `market_data:yahoo:invalid:<symbol>` (separate from the price-key prefix the Java reader consumes) with a 24h TTL; a successful price write atomically `DELETE`s the matching invalid marker in the same Redis pipeline, so the negative cache self-recovers on the next successful fetch. `/force-fetch` returns 404 for REJECTED (with `mark_invalid`) and 503 for FAILED (no negative-cache write), giving the Java backend a typed retry signal — 404 means stop, 503 means back off and retry. The Python sweeper decrypts the `fyers_token.access_token` ciphertext with the same AES-256-GCM key the Java `EncryptedStringConverter` writes (`Base64(IV(12) ‖ ct ‖ tag(16))`, `ENCRYPTION_KEY` env var must be byte-identical to backend); token-error codes invalidate the in-process token cache. Composes with ADR-040 (which filters known-bad categories upstream at SQL) as the second line of defence for residual per-call rejections (ADR-042).

## 5. UI/UX Identity & Styling

- **Full-bleed workspace canvases** — dashboards/holdings use the full viewport width; no framing gutters.
- **Tabular-nums for all financial data** — monetary/percentage values use `font-variant-numeric: tabular-nums` and are **right-aligned**.
- **Strict contrast floors** — primary text never lighter than `text-gray-500`; secondary-line reporting-currency rows use smaller, muted weight but stay readable.
- **Stacked multi-currency cells** — native-currency value as primary line, reporting-currency beneath (smaller/lighter). Secondary line hidden when `holding.currency === reportingCurrency` (zero-noise rule).
- **Progressive FX disclosure** — ⓘ icon in Gain/Loss cell, tooltip shows asset return vs. FX impact. Inline cost is zero.
- **Gain/Loss column is reporting-currency only** — enforces tie-out with summary header.
- **Modern Angular control flow only** — `@if`/`@for`/`@switch`; OnPush everywhere; Signals for state, `async` pipe in templates.
- **ECharts for data viz** — declarative `EChartsOption` via Signal; pad-angle donuts, border-radius, emphasis animations. No Chart.js.
- **BEM-inspired SCSS** with design tokens; no repeated raw hex. Use the frontend-design plugin for premium design work (DESIGN.md is deprecated).
- **Hybrid zero-states** — search surfaces preserve the affordance *and* inject actionable live data (e.g., Portfolio Movers) instead of empty icons.

## 6. Security & Boundary Constraints

- **Never log PII** — user emails, broker credentials, raw transaction payloads stay out of logs.
- **Sensitive fields AES-256 encrypted** via `SecretEncryptionService` (SnapTrade OAuth creds, API tokens).
- **JWT bearer on every request** — frontend `AuthInterceptor` injects; `authGuard` protects routes.
- **Single-origin deployment** — Caddy path-routes `/api/*` → backend, `/*` → frontend. No CORS; no third-party cookie domains.
- **Container network isolation** — only Caddy exposes 80/443. Postgres/Redis/price-sweeper/backend are reachable only on `app-network`.
- **`price-sweeper /force-fetch` is internal-only** — never route through Caddy.
- **Never share Postgres credentials beyond the compose network** — the sidecar reads the same DB; schema changes require coordinated review.
- **SnapTrade lookups use slugs**, not user-visible identifiers; account IDs stored as JSONB on the connection, not leaked to the client.
- **No client-side financial aggregation** — totals, realized gain, FX conversion, and tie-out math are always server-computed. The frontend renders server-authoritative DTOs.
- **Admin endpoints (FX backfill, reporting-currency rebuild, vendor-token capture) are superuser-gated** and bounded (e.g., 365-day max per historical-FX call). System-identity vendor tokens (Fyers V3, future siblings) are captured exclusively via `POST /api/v1/admin/<vendor>/exchange` and stored as encrypted singletons; the response carries metadata only (ADR-036).
- **Flyway owns schema** — Hibernate runs in `validate`; no runtime DDL. Migrations are idempotent (`IF NOT EXISTS`, `ON CONFLICT`) and backward-compatible.
- **No secrets in repo or chat** — credentials come from environment variables (Pydantic Settings on Python, Spring config on Java).
